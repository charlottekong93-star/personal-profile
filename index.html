Podcast Producer Agent

— 一个承担播客制作 PMO 角色的 Agent
Problem
我已经用 AI 完成剪辑标注、Shownote、标题和发布复盘，但这些能力彼此分散。每制作一期播客，仍然需要我记住当前进度、准备对应材料、判断下一步，再逐个调用不同的 AI 工具。AI 提升了单项任务的效率，但负责顾全整个制作流程的人仍然是我。
Action
我把原本分散的播客 Skills 重构成 Podcast Producer Agent。

How it works —— 它不只执行单项任务，还承担播客制作 PMO 角色：记录每期节目当前进行到哪一步，判断下一步应该做什么，自动调用剪辑标注、Shownote 或复盘等专项能力，并检查结果是否完成。

我在其中的角色 —— 选题、听感精剪、最终标题、内容质量和外部发布等需要创作判断的环节，Agent 会暂停并把决定权交还给我。

P · 03
Tool Calling Agent 行为设计

飞书 AI Agent

— 为真实协作场景设计的 Agent
Problem
飞书是我的主要协作环境，但通用 AI 无法直接读取或操作真实的文档、多维表格、群聊和日程；仅给出文字建议，也无法确认任务是否真的在系统中完成。
Action
系统连接 —— 使用 Node.js、DeepSeek API 和 lark-cli 搭建 Bridge Server，接入文档、多维表格、群聊、日程等 15 个工具，让 Agent 能在真实飞书环境中读取和执行任务。

行为控制 —— 用 System Prompt 约束任务拆解、工具调用和异常处理；用摘要式长期记忆保留个人偏好和历史决定；通过工具权限限制高风险操作，保留必要的人工确认。
Result
一次修改多维表格名称时，Agent 曾声称操作成功，但系统中的名称并未正确更新。后来我为写入工具增加「执行后回读」机制：操作完成后重新读取飞书中的真实状态，只有结果与目标一致时才确认成功，否则继续检查或重试。
