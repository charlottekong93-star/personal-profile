P · 01
Workflow Skill Agent
2025

播客内容 AI 系统 — Skill 与 Agent 的两层设计

Problem
播客后期最卡人的不是录制，是写 Shownote、定标题、做剪辑标注 —— 这些步骤既耗时间，又最容易因为状态不好而输出不稳定。复盘也容易被拖延或遗漏。
Action
分两层设计：

Workflow Skill 层 —— 把剪辑标注、Shownote 生成、标题撰写、发布复盘拆解为四个结构化 Skill，上传逐字稿后按顺序调用，每步输出直接喂入下一步。

数据追踪 & 内容复盘 Agent —— 持续监听播客 RSS，检测到新集自动拉取数据、在 Obsidian 生成档案草稿，并通过 AppleScript 设定 48h 和 7 天两次复盘提醒，全程不需要手动触发。
Result
制作时间缩短 50% 以上。更重要的是：输出质量不再因个人状态波动，每期风格稳定可预期；复盘也不再遗漏。
→ 可展示：Before / After 对比截图，或一个示例输出
P · 02
Personal OS Automation
2025 · Jan – Now

个人 AI 操作系统 — Personal OS

Problem
Gap 期最怕靠意志力维持系统 —— 状态一差，什么都垮。需要一套能自动运转、不依赖我每天手动推动的个人系统。
Action
系统经历了两个阶段：1 月用 Notion 搭起框架，确立 PARA 方法的知识管理结构；4 月升级为 Claude Code + Obsidian，把播客工作流的关键节点自动化 —— RSS 监听检测新集 → 自动生成档案草稿 → macOS 设定复盘提醒，通过 Python + cron + AppleScript 实现无人值守运转。
Result
整体框架稳定运行 5 个月；自动化系统（Claude Code + Obsidian）自 4 月上线以来已持续运行 1 个月。播客复盘流程不再依赖手动触发，每日启动稳定。
→ 可展示：系统架构图（一张图说清楚整个框架）
P · 03
Self-built Agent
2025

飞书 AI 机器人 — 自建 Agent

Problem
飞书是我和共创伙伴的主要协作平台，但没有一个真正懂我工作流的 AI 助手 —— 通用 AI 不理解飞书的结构，飞书自带的功能又不够灵活。
Action
用 Node.js + DeepSeek API + lark-cli 搭建本地 Bridge Server，把飞书聊天和 AI 能力连接起来，接入文档读写、多维表格管理、日程查询等 8 个飞书工具。通过 System Prompt 设计让它具备 Agent 行为：主动拆解任务、遇到异常自己处理。
Result
机器人持续运行中。现在可以直接在飞书对话里管理播客内容表格、新建文档、更新记录，不需要切换任何界面。
