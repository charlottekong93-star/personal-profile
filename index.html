P · 02
Workflow Skill 上下文设计

播客内容 AI 系统

— Workflow Skill + 追踪 & 复盘 Agent
Problem
播客后期包含剪辑标注工作稿、Shownote、标题和发布复盘，多项任务前后依赖，又需要保留内容判断。每期只把逐字稿交给通用 AI，不仅要反复解释节目背景，输出质量也容易随提示方式和个人状态波动。
Action
上下文 —— 将节目定位、目标听众、主持人表达风格和历史优秀 Shownote 作为长期背景；每期再加入本期逐字稿、时间戳和当前制作阶段，让 AI 知道节目在对谁说话、应该如何表达，以及任务进行到哪一步。

Workflow —— 把剪辑标注工作稿、Shownote、标题、发布复盘拆成 4 个 Skill。每个 Skill 都有明确的输入、输出和人工确认节点；确认后的结果可以进入下一步，不需要每期从头解释。发布后，追踪 & 复盘 Agent 监听 RSS，生成档案草稿并设置复盘提醒。

质量控制 —— 不添加逐字稿中不存在的观点；观点和时间戳必须能回溯原文；内容取舍与最终表达保留人工判断。人工确认后的修改结果会继续用于调整 Skill，让下一次输出更贴近节目标准。
Result
一份 33 分钟、5000+ 字的精剪逐字稿，约 3 分钟生成结构化 Shownote 初稿；整体制作时间减少 80% 以上，内容校验和发布复盘成为默认流程。

P · 03
Tool Calling Agent 行为设计

飞书 AI Agent

— 为真实协作场景设计的 Agent
Problem
飞书是我的主要协作环境，但通用 AI 无法直接读取或操作真实的文档、多维表格、群聊和日程；仅给出文字建议，也无法确认任务是否真的在系统中完成。
Action
系统连接 —— 使用 Node.js、DeepSeek API 和 lark-cli 搭建 Bridge Server，接入文档、多维表格、群聊、日程等 15 个工具，让 Agent 能在真实飞书环境中读取和执行任务。

行为控制 —— 用 System Prompt 约束任务拆解、工具调用和异常处理；用摘要式长期记忆保留个人偏好和历史决定；通过工具权限限制高风险操作，保留必要的人工确认。
Result
一次修改多维表格名称时，Agent 曾声称操作成功，但系统中的名称并未正确更新。后来我为写入工具增加「执行后回读」机制：操作完成后重新读取飞书中的真实状态，只有结果与目标一致时才确认成功，否则继续检查或重试。
